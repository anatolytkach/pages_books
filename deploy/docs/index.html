ReaderPub: передача проекта

Документация на русском языке. Актуально на 2026-03-30.

Версия страницы: docs-live-2026-03-30-catalog-grouped-categories

1. Назначение и границы

Проект публикует каталог и читалку EPUB: /books/ и /books/reader/. Контент книг хранится в R2 как распакованные EPUB-папки, а поиск/навигация строятся по JSON-индексам, которые генерируются локально и загружаются в R2.

Репозиторий проекта для разработчика: https://github.com/anatolytkach/pages_books.git 

git clone https://github.com/anatolytkach/pages_books.git
cd pages_books

2. Все worker'ы Cloudflare

Worker A (Pages worker): _worker.js (Pages worker проекта reader-books).

  • /books/api/* -> чтение индексов из R2 binding READER_BOOKS.
  • /books/api/translate, /api/translate -> Google Translate API.
  • /books/api/notes-share*, /api/notes-share*, /books/api/ns*, /api/ns*.
  • /books/ping, redirect /books/<id> -> /books/reader/#<id>.
  • SSR/edge-rendering SEO pages: /book/*, /author/*, /category/*, /sitemap.xml, /sitemaps/*, /robots.txt.

Worker B (служебный notes proxy): tools/runtime/notes-share-proxy-worker.js.

  • KV storage заметок: ключи ns:<shareId> в READERPUB_NOTES_SHARE_KV.
  • Routes: /books/api/notes-share*, /api/notes-share*, алиасы /ns*.

Worker C (docs gateway): tools/runtime/docs-route-worker.js, deployed как readerpub-docs-route.

  • Routes: reader.pub/docs, reader.pub/docs/*.
  • Назначение: Basic Auth + проксирование docs на master.reader-books.pages.dev/docs/*.
  • Правило публикации: изменения в docs/README.md нужно сразу переносить в deploy/docs/index.html и публиковать через tools/dev/deploy_docs.sh; этот скрипт всегда выкатывает docs в ветку master, потому что staging.reader.pub/docs/ читает docs именно оттуда.

Worker D (production router): tools/runtime/reader-books-router.js.

  • Routes: /books*, /book*, /author*, /category*, /sitemap.xml*, /sitemaps/*, /robots.txt*.
  • Читает R2 binding BOOKS напрямую для api/*, content/* и seo/*.
  • На production именно этот router связывает root SEO routes с runtime logic Pages worker.

3. Структура хранения в R2

  • Книги: content/<book_id>/... (распакованный EPUB, включая META-INF/container.xml).
  • Индексы: api/...json.
  • Основные файлы: api/letters.json, api/languages.json.
  • Навигация по авторам: api/p/*.json, api/a/*.json.
  • Поиск: api/search/<token3>.json.
  • Языковые ветки browse-навигации: api/lang/<lang>/letters.json, api/lang/<lang>/p/*.json, api/lang/<lang>/a/*.json.
  • SEO data layer: seo/book-shards/*.json, seo/author-shards/*.json, seo/category/*.json, seo/sitemaps/*.json, seo/version.json.

4. SEO-layer: архитектура и runtime

SEO-layer добавлен поверх существующего каталога и веб-читалки. Он не переписывает /books/ и не заменяет reader, а даёт отдельные индексируемые URL и серверный HTML для поисковиков.

Публичные SEO routes:

  • /book/<slug>
  • /book/<slug>/chapter-<n>-<chapter-slug>
  • /author/<slug>
  • /category/<slug>
  • /sitemap.xml, /sitemaps/*, /robots.txt

Как это работает:

  • HTML SEO pages не предгенерируются в Pages.
  • Book/chapter/author/category pages рендерятся on demand в _worker.js.
  • Source data для SEO pages читаются из R2, а не из Pages assets.
  • Pages bundle содержит только код и frontend assets; SEO manifests вынесены в R2.

Source of truth:

  • content/<book_id>/... — XHTML/OPF/обложки книг.
  • api/* — индексы каталога /books/.
  • seo/* — manifests для SEO routes и sitemap chunks.

Book manifest хранит:

  • id, slug, title
  • authorName, authorSlug
  • categories[], chapters[]
  • description, meta_description
  • raw_description, normalized_description, description_source

Description pipeline:

  • Сначала пробуется metadata description из EPUB.
  • Потом — осмысленный fragment из первой нормальной линейной главы.
  • Если хороший snippet не найден, используется безопасный fallback вида Read "Title" by Author on ReaderPub.
  • Нормализация удаляет Gutenberg boilerplate, frontmatter, TOC noise и служебные opening fragments.

Внутренние ссылки и UX:

  • Open in WeRead ведёт в существующий flow /books/<id>/.
  • Category chips на SEO book page ведут в каталог: /books/#view=category&category=<slug>.
  • SEO category page /category/<slug> остаётся отдельной индексируемой страницей и дополнительно содержит CTA Open in Catalog.

Sitemap и cache:

  • /sitemap.xml собирается из seo/sitemaps/index.json.
  • SEO pages отдают x-reader-seo-version; cache key включает version из seo/version.json.
  • При rebuild manifests новая version автоматически создаёт новый namespace edge cache.

5. Форматы индексов каталога

Источник: tools/catalog/build_lang_indexes.py.

  • letters.json: список букв/ключей и количества авторов.
  • p/<node>.json: prefix-tree. Либо authors, либо prefixes, плюс authorCount.
  • a/<author_key>.json: автор и массив его книг (id/title/cover).
  • search/<token3>.json: mixed items по токену: авторы (t:"a",k,n,c) и книги (id,title,a,k,cover).
  • languages.json: список языков и количества книг.

Языкового search-слоя больше нет: поиск всегда использует глобальные api/search/<token3>.json, а языковые ветки сохраняют только browse-навигацию по буквам, префиксам и авторам.

Ограничения и правила:

  • max_prefix по умолчанию 5;
  • threshold по умолчанию 50 (решает, когда хранить authors, а когда дробить на prefixes);
  • поиск bucket'изируется по первым 3 символам нормализованного слова;
  • книги индексируются по значимым словам заголовка, авторы — по значимым словам author index name;
  • для книг и авторов исключаются stop-words: the, a, an, and, or, of, to, in, on, for, by;
  • для книг дополнительно исключаются service-words: vol, volume, no, part, chapter.

Frontend UX:

  • search требует минимум 3 символа;
  • frontend search использует token-based matching по словам запроса, а не glued-string matching;
  • порядок слов в запросе не важен для match'а (william sha и sha william работают одинаково);
  • склеенные запросы без разделителей вроде williamsha не считаются multi-word match;
  • search results сортируются по алфавиту: сначала authors, затем titles;
  • search results рендерятся постранично по 12 элементов на страницу;
  • pager search results показывается и сверху, и снизу списка, а переключение страницы делает мягкий scroll к началу search-блока;
  • author books и category books используют обычный pager Showing X-Y of Z, без Show more.

Текущее UI-поведение каталога /books/:

  • Header brand: WeRead с subtitle Powered by ReaderPub; рядом с ним находятся меню Newest, Popular, By Author, By Category, search и My Books.
  • На узких экранах header использует гамбургер-меню для навигации, а search остается отдельной строкой под первой строкой header.
  • Mobile hamburger раскрывается на всю ширину и перекрывает search, а не раздвигает layout.
  • Переход через верхние меню скроллит так, чтобы заголовок секции оставался видимым; активный menu item временно становится черным и некликабельным до следующего ручного scroll.
  • Hero состоит из CTA-карточки слева и изображения readers справа; CTA раскрывает dropdown Choose a book to try in WeRead с 5 книгами и не делает autostart.
  • Landing order: Newest Releases → Popular Titles → Browse by Category → Browse by Author.
  • Browse by Category показывает 9 Gutenberg-групп; каждая группа — отдельная dropdown-выкидушка поверх страницы, а внутри нее алфавитный список категорий. Длинные labels переносятся на новую строку.
  • Browse by Author собран в одну card-like секцию; внутри нее находятся title, language selector, breadcrumbs, reset-control и динамический browse content.
  • Клики по буквам/префиксам/авторам не должны убирать остальные landing sections; обновляется только внутренний browse content этого блока.
  • Если пользователь ушел внутрь browse-state (prefixes, prefix, author), справа в заголовке Browse by Author появляется undo/reset-иконка, которая возвращает секцию в исходный state без прыжка по странице.
  • Все книжные секции используют единые card-style карточки: счетчик N books стоит справа в строке заголовка, книги пагинируются по 12, названия темно-зеленые и underline показывается только на hover.
  • При активном search hero и остальные catalog sections скрываются, на экране остаются только search results; очистка query возвращает обычный landing layout.
  • My Books view внутри каталога скрывает hero и Browse by Author, показывает локальный список даже без Google Drive integration и держит Back to Catalog в header.
  • Обычные book-open links внутри каталога используют короткий micro-transition book-opening на 180ms; modified clicks и hero/demo links не перехватываются.

Entry semantics для входа в reader:

  • entry=hero используется только для hero/demo preview и не должен считаться реальным пользовательским чтением.
  • Для entry=hero не сохраняются readerpub:lastid, readerpub:lastsource, readerpub:lastcfi:<book_id>; такая книга не должна попадать в My Books ни локально, ни через Drive sync snapshot.
  • entry=catalog добавляется к обычным открытиям книг из общего каталога.
  • entry=mybooks используется для открытий книги из My Books в каталоге и из My Books внутри reader.
  • Reader onboarding hints показываются только для entry=catalog.

6. Конвейер DOCX - EPUB - R2 - индексы - Pages

Скрипты:

  • books/content/make_epub_from_docx.sh — сборка EPUB из DOCX + нормализация.
  • books/content/gen_epub_css_from_docx.py — извлечение heading-стилей Word в CSS.
  • books/content/epub_unpack.sh — распаковка EPUB в папки книг (новые id или replace).
  • books/content/epub_publish.sh — upload в R2 + переиндексация + Pages deploy.
  • tools/catalog/build_lang_indexes.py — генератор индексов.

Сборка EPUB:

cd books/content
./make_epub_from_docx.sh ru "Название книги" "Имя Автора"

PDF - EPUB (reflowable):

cd books/content
./make_epub_from_pdf.sh en "Book Title" "Author Name"
  • make_epub_from_pdf.sh использует pdftohtml + pandoc и чистит PDF-артефакты (headers/footers/page numbers).
  • По умолчанию включена защита от дублирования страниц картинками: STRIP_SYNTHETIC_PAGE_IMAGES=1.
  • Обложка берется из books/content/cover.jpg или переменной COVER_IMAGE.

Распаковка EPUB:

cd books/content
./epub_unpack.sh import-all
./epub_unpack.sh replace 77752 /path/to/book.epub

Публикация в R2 и каталог:

cd books/content
./epub_publish.sh upload-ids 77752 77753
./epub_publish.sh upload-ids 77752 77753 --no-image-upload
./epub_publish.sh reindex-ids 77752
  • --no-image-upload — не перезаливает изображения книги в R2; старые картинки остаются как есть, обновляются текст/структура и каталог.

6.1 Workflow: безопасный production deploy каталога и читалки

Для UI-изменений в books/, reader/ и _worker.js production deploy нужно делать отдельно от контентного конвейера.

  • production branch у Cloudflare Pages проекта reader-books называется production, а не master;
  • ветка master у этого проекта создает только preview deploy и не обновляет reader.pub/books/*;
  • production router tools/runtime/reader-books-router.js проксирует reader.pub/books/reader/* в https://reader-books.pages.dev/reader/*, поэтому для боевого обновления должен обновиться именно production alias reader-books.pages.dev.

Правильный порядок:

  1. Все изменения сначала проверять на локальном сайте; не выкатывать их сразу на production без явного подтверждения пользователя после локальной проверки.
  2. Собрать минимальный deploy bundle, а не деплоить весь корень репозитория.
  3. Включать в bundle только _worker.js, books/ без books/content/, reader/ и publisher_tasks/ (worker импортирует ./publisher_tasks/service.mjs).
  4. Не включать тяжелые артефакты вроде reader_seo_indexes/, иначе Pages может отклонить deploy из-за лимита 25 MiB на файл.
  5. Деплоить Pages project reader-books в branch production только после явной команды пользователя на production deploy.
wrangler pages deploy /tmp/readerpub_deploy \
  --project-name reader-books \
  --branch production \
  --commit-dirty=true

Проверка после deploy:

  1. wrangler pages deployment list --project-name reader-books — новый deploy должен появиться как Environment = Production, Branch = production.
  2. https://reader-books.pages.dev/reader/ — upstream alias должен отдавать новую версию.
  3. https://reader.pub/books/reader/ — боевой URL должен отдавать ту же версию.

Чего не делать:

  • не выкатывать изменения на production автоматически сразу после правок;
  • не деплоить UI-изменения в branch master, если нужен production;
  • не считать, что текущая git-ветка репозитория совпадает с production branch в Cloudflare Pages;
  • не деплоить из корня репозитория без проверки состава файлов.

6.2 Workflow: git commit и push

Правило для работы в этом проекте:

  • все пользовательские команды на коммит нужно выполнять по логике ./commit_logic.sh;
  • commit_logic.sh обязан стадить и код проекта, и документацию, если в ней есть изменения, затем делать git commit и сразу git push в remote;
  • поэтому команда пользователя комит / сделай коммит для этого проекта означает не только локальный commit, но и отправку коммита в remote;
  • если меняется логика коммитов, source-of-truth находится в commit_logic.sh, и workflow нужно выравнивать под него, а не наоборот.

Проверка:

./commit_logic.sh "commit message"
git status --short --branch

Ожидаемый результат после успешного push:

  • ветка не должна оставаться в состоянии ahead N.

7. Ручной Gutenberg CLI

Для новых книг из Project Gutenberg weekly automation больше не используется. Вместо нее проект использует ручной terminal-first pipeline: tools/gutenberg/gutenberg_manual_ingest.py.

Зачем он нужен:

  • сам находит старший Gutenberg ID в текущей системе;
  • сам начинает со следующего номера, без ручного указания from-id;
  • скачивает только preferred EPUB-вариант pg<ID>.epub;
  • заливает новые книги в legacy Gutenberg layout content/<id>/...;
  • обновляет каталоговые индексы, Newest Releases и selective SEO;
  • показывает прогресс в терминале и умеет resume после сбоя.

Bulk upload и cleanup:

  • для массовой загрузки content/* и api/* CLI предпочитает rclone, если доступен remote r2 или явно передан --rclone-remote;
  • fallback остается прежним: прямой R2 S3 доступ или wrangler;
  • после rebuild CLI удаляет устаревшие lang/*/search локально и на R2.

Каноническая модель после ingest:

  • Gutenberg-книги остаются в reader как ?id=<gutenberg_id> без source=gutenberg.
  • Manual-книги остаются source-qualified: ?id=<manual_id>&source=manual.
  • Newest Releases строится по catalog_added_at за последние 30 дней, без hard limit по количеству книг.
  • Category taxonomy сохраняет Gutenberg grouping metadata: groupTitle, groupSlug, groupOrder, а summary payload может содержать top-level groups[].

Основные команды:

python3 tools/gutenberg/gutenberg_manual_ingest.py status
python3 tools/gutenberg/gutenberg_manual_ingest.py scan
python3 tools/gutenberg/gutenberg_manual_ingest.py run
python3 tools/gutenberg/gutenberg_manual_ingest.py resume

Дополнительные runtime flags:

python3 tools/gutenberg/gutenberg_manual_ingest.py run --rclone-remote r2
python3 tools/gutenberg/gutenberg_manual_ingest.py run --skip-rclone

Что делает каждая команда:

  • status — показывает текущий max Gutenberg ID, следующий стартовый ID и состояние текущей сессии.
  • scan — ищет новые Gutenberg-книги без импорта.
  • run — выполняет полный импорт: книги, индексы, Newest Releases, SEO.
  • Во время этого же цикла обновляется Gutenberg category taxonomy, поэтому новые книги после ingest автоматически попадают в соответствующие категории и в корректные category groups.
  • resume — продолжает незавершенный run после сбоя.

Пример безопасного рабочего порядка:

  1. python3 tools/gutenberg/gutenberg_manual_ingest.py status
  2. python3 tools/gutenberg/gutenberg_manual_ingest.py scan
  3. python3 tools/gutenberg/gutenberg_manual_ingest.py run
  4. если run оборвался — python3 tools/gutenberg/gutenberg_manual_ingest.py resume

Пошаговая инструкция для работы в терминале:

  1. Откройте терминал в корне репозитория: 
    cd /Volumes/2T/se_ingest/pages_books
  2. Проверьте текущее состояние pipeline: 
    python3 tools/gutenberg/gutenberg_manual_ingest.py status
    Ожидаемо увидеть:
    • max_gutenberg_id — текущий старший Gutenberg ID в системе;
    • next_start_id — номер, с которого начнется следующий импорт;
    • current_run_status — состояние текущей сессии, если она уже была.
  3. Сделайте безопасный предварительный поиск новых книг без импорта: 
    python3 tools/gutenberg/gutenberg_manual_ingest.py scan
    После этого:
    • CLI сам найдет старший Gutenberg ID;
    • CLI сам вычислит следующий стартовый ID;
    • в выводе появится число новых кандидатов.
  4. Если нужно заранее проверить, что у кандидатов реально есть preferred EPUB-вариант, используйте: 
    python3 tools/gutenberg/gutenberg_manual_ingest.py scan --verify-epub-on-scan
    Этот режим не импортирует книги, а только проверяет наличие pg<ID>.epub.
  5. Если кандидаты найдены и запуск одобрен, запустите полный импорт: 
    python3 tools/gutenberg/gutenberg_manual_ingest.py run
    Во время выполнения CLI последовательно делает:
    • скачивание EPUB;
    • распаковку;
    • загрузку в /books/content/<id>/;
    • обновление каталоговых индексов;
    • обновление Newest Releases по окну последних 30 дней, без hard limit;
    • селективную генерацию и публикацию SEO.
  6. Следите за прогрессом прямо в терминале. Основные фазы, которые должны появляться в логе:
    • download
    • unpack
    • upload_content
    • index_catalog
    • newest
    • seo_build
    • seo_upload
    Параллельно локальный лог пишется в: 
    /tmp/readerpub_gutenberg_runs/<timestamp>-run.log
  7. Если run завершился успешно, еще раз проверьте итог: 
    python3 tools/gutenberg/gutenberg_manual_ingest.py status
    Затем проверьте каталог и reader вручную:
    • поиск новой книги в /books/;
    • открытие книги в reader как ?id=<gutenberg_id>;
    • наличие новых книг в Newest Releases;
    • наличие новых SEO book pages.
  8. Если run оборвался, не запускайте новый run поверх старого состояния. Вместо этого продолжайте текущую сессию: 
    python3 tools/gutenberg/gutenberg_manual_ingest.py resume
    CLI должен продолжить с незавершенной книги и незавершенной фазы.
  9. Если книги уже импортированы, но SEO по какой-то причине не дописалось, используйте отдельный добор SEO: 
    python3 tools/gutenberg/gutenberg_manual_ingest.py seo
  10. Если нужно только пересобрать витрину Newest Releases, без нового импорта книг: 
    python3 tools/gutenberg/gutenberg_manual_ingest.py newest

Прогресс и логи:

  • CLI пишет фазовый progress в терминал: download, unpack, upload_content, index_catalog, newest, seo_build, seo_upload.
  • Локальный run-log создается в /tmp/readerpub_gutenberg_runs/<timestamp>-<command>.log.
  • Состояние pipeline хранится в R2: system/gutenberg-pipeline/state.json.

Поведение при сбое:

  • если ошибка временная, pipeline пытается пережить ее автоматически;
  • если ошибка hard-failure, CLI пишет книгу, фазу и точную команду продолжения;
  • основная команда восстановления: python3 tools/gutenberg/gutenberg_manual_ingest.py resume.

Runtime credentials:

  • предпочтительный режим: прямой R2 S3 доступ через R2_S3_ENDPOINT, R2_ACCESS_KEY_ID, R2_SECRET_ACCESS_KEY;
  • bulk-sync режим: rclone remote r2 или явный --rclone-remote;
  • fallback: активная wrangler session / CLOUDFLARE_API_TOKEN.

7.5 Gutenberg updater

Скрипт: tools/gutenberg/update_gutenberg_catalog.py.

Назначение:

  • weekly/queued ingest новых Gutenberg-книг без ручного from-id;
  • массовая загрузка контента и каталоговых индексов;
  • rebuild Newest Releases и selective SEO;
  • синхронизация новой схемы индексов без api/lang/<lang>/search/*.

Особенности:

  • для bulk upload предпочитает rclone, если доступен remote r2;
  • можно явно передать --rclone-remote или отключить rclone через --skip-rclone;
  • после rebuild удаляет устаревшие lang/*/search локально и purge-ит api/lang/*/search на R2.

Детальная инструкция для разработчика: docs/gutenberg-pipeline.md.

7.6 Reader1 publish pipeline

Новый pipeline для source-qualified книг в новом формате: tools/reader1/publish_books.py.

Назначение:

  • импортировать EPUB сразу в новый reader1-формат без промежуточной legacy-распаковки;
  • поддерживать произвольный source вроде manual, partner, archive, а не только manual;
  • публиковать книги так, чтобы каталог мог потом выбирать читалку по явному readerType: "reader1";
  • пересобирать все нужные индексные слои каталога: a/*, p/*, search/*, letters.json, lang/*, book-locations/*, newest;
  • поддерживать status и resume после сбоя, как в Gutenberg pipeline.

Критичные правила:

  • этот pipeline предназначен только для non-Gutenberg книг;
  • source=gutenberg здесь запрещен намеренно;
  • Gutenberg-книги по-прежнему заливаются только через legacy Gutenberg pipeline и остаются в root без source;
  • вызывать pipeline нужно из корня репозитория:
cd /Volumes/2T/se_ingest/pages_books

Что он делает в полном цикле:

  1. принимает один EPUB, папку с EPUB или zip-архив с EPUB;
  2. staging-ит исходные EPUB в локальную очередь;
  3. назначает reader_id и source_book_id;
  4. конвертирует EPUB в reader1-формат прямо в books/content/<source>/<source_book_id>/;
  5. пересобирает каталоговые индексы и book-locations;
  6. обновляет Newest Releases;
  7. валидирует, что книга реально появилась во всех обязательных слоях;
  8. при --publish загружает content и изменившиеся api/* на боевое хранилище.

Основные команды:

python3 tools/reader1/publish_books.py status
python3 tools/reader1/publish_books.py run --source manual --epub /path/to/book.epub
python3 tools/reader1/publish_books.py run --source manual --input-dir /path/to/folder
python3 tools/reader1/publish_books.py run --source manual --input-zip /path/to/books.zip
python3 tools/reader1/publish_books.py resume

Поддерживаются и совместимые алиасы:

python3 tools/reader1/publish_books.py publish-epub /path/to/book.epub --source manual
python3 tools/reader1/publish_books.py publish-dir /path/to/folder --source manual
python3 tools/reader1/publish_books.py publish-zip /path/to/books.zip --source manual

Какой режим когда использовать:

  • status — перед запуском и после него; показывает текущее состояние pipeline, активный run, текущую книгу, фазу, число успехов и ошибок;
  • run --epub — одна новая книга;
  • run --input-dir — пакет из нескольких EPUB, уже лежащих в одной папке;
  • run --input-zip — пакет из zip-архива, если EPUB приходят одним архивом;
  • resume — продолжение незавершенного run после исправления ошибки.

Обязательные и основные параметры:

  • --source <name> — обязательный source для всех non-Gutenberg книг; именно он определяет, куда книга ляжет и как будет квалифицирована в каталоге;
  • --epub <path> — путь к одному EPUB;
  • --input-dir <path> — путь к папке с несколькими EPUB;
  • --input-zip <path> — путь к zip-архиву, внутри которого лежат EPUB;
  • --source-book-id <id> — явный публичный id внутри source; обычно задается только для одиночной книги, если нужен конкретный номер;
  • --start-source-book-id <id> — стартовый публичный id для batch-run; полезно, когда пакет книг должен занять заранее выбранный диапазон;
  • --label <text> — человекочитаемая подпись импорта в source_registry.json;
  • --publish — после локальной сборки и валидации сразу загрузить content и api/* на продовое хранилище;
  • --dry-run — прогнать pipeline без фактической remote upload-фазы;
  • --skip-rclone — не использовать rclone даже если он доступен;
  • --rclone-remote <name> — явно задать remote для bulk upload;
  • --newest-window-days <N> — окно для Newest Releases, по умолчанию 30;
  • --newest-max-books <N> — hard limit для Newest Releases; 0 означает без лимита.

Служебные параметры, которые обычно не трогают:

  • --content-root — локальный content-root, по умолчанию books/content;
  • --index-root — локальный индексный root, по умолчанию reader_lang_indexes;
  • --registry — путь к tools/state/source_registry.json;
  • --state-file — путь к state-файлу pipeline;
  • --queue-root — локальная папка очереди для staged EPUB;
  • --python-bin, --wrangler-bin, --rclone-bin, --state-r2-bucket — runtime overrides для нестандартного окружения.

Пошаговый безопасный порядок работы:

  1. Открыть терминал в корне репозитория: 
    cd /Volumes/2T/se_ingest/pages_books
  2. Проверить текущее состояние: 
    python3 tools/reader1/publish_books.py status
  3. Запустить локальный import без публикации: 
    python3 tools/reader1/publish_books.py run --source manual --epub /absolute/path/to/book.epub
  4. Проверить локально:
    • появился ли content в books/content/<source>/<source_book_id>/;
    • появились ли нужные записи в reader_lang_indexes;
    • открывается ли книга в локальном каталоге и в reader1.
  5. Только после локальной проверки повторить тот же запуск с --publish, либо запустить новый batch с --publish.

Примеры рабочих запусков:

Одна книга:

python3 tools/reader1/publish_books.py run \
  --source manual \
  --epub /absolute/path/to/book.epub \
  --source-book-id 20

Несколько EPUB из папки:

python3 tools/reader1/publish_books.py run \
  --source manual \
  --input-dir /absolute/path/to/epub-folder

Несколько EPUB из zip:

python3 tools/reader1/publish_books.py run \
  --source manual \
  --input-zip /absolute/path/to/books.zip

Публикация после локальной проверки:

python3 tools/reader1/publish_books.py run \
  --source manual \
  --epub /absolute/path/to/book.epub \
  --publish

Где pipeline хранит свое состояние:

  • state-файл: tools/state/reader1_publish_state.json
  • очередь staged EPUB для текущих run: tools/state/reader1_publish_queue/<run_id>/sources/...
  • локальные логи: /tmp/reader1_publish_runs/<timestamp>-<command>.log

Что означает status:

  • current_run_id — идентификатор текущего или последнего незавершенного run;
  • current_run_source — source текущего run;
  • current_run_input_mode — epub, dir или zip;
  • current_run_book — текущая книга по внутреннему reader_id;
  • current_run_phase — текущая фаза;
  • current_run_success / pending / failed — статус книг внутри run;
  • current_run_log — путь к локальному log-файлу.

Основные фазы:

  • convert
  • upload_content
  • index_catalog
  • book_locations
  • newest
  • validate
  • upload_api

Как реагировать на ошибки и что делать в диалоге:

  • если run завершился без ошибок, следующий шаг — локальная проверка или ручная проверка продового каталога после --publish;
  • если ошибка возникла до публикации, не запускать новый batch поверх старого состояния; сначала посмотреть python3 tools/reader1/publish_books.py status и лог из /tmp/reader1_publish_runs/...;
  • если ошибка понятна и исправлена, продолжать только через python3 tools/reader1/publish_books.py resume;
  • если ошибка связана с неправильным EPUB, лучше сначала исправить источник, удалить неудачный локальный output этой книги и потом повторить запуск осознанно, а не накапливать новые run поверх поврежденной книги;
  • если ошибка связана с remote upload, после исправления credentials или сети использовать resume, потому что staged EPUB уже сохранены в queue;
  • если status показывает pending_retry, это означает, что run надо именно продолжать, а не начинать новый run.

Правила использования:

  • не использовать этот pipeline для Gutenberg;
  • не смешивать в одном run книги разных source;
  • не публиковать сразу большой пакет без локального прогона хотя бы на одной книге;
  • при batch-импорте не задавать --source-book-id для нескольких книг; для batch использовать либо автоназначение, либо --start-source-book-id;
  • content-файлы книг не коммитить в git;
  • для каталога source-книги должны попадать в индексы с явным readerType: "reader1", и этот pipeline это делает сам.

8. Операционные сценарии и rollback

replace <id> — безопасный способ исправить одну книгу.

  • Кладете ровно один EPUB в books/content/.
  • Запускаете ./epub_unpack.sh replace <id>.
  • Затем ./epub_publish.sh upload-ids <id>.

Rollback:

  • повторный replace <id> предыдущим EPUB;
  • или upload-ids для восстановления папки книги + reindex-ids через epub_publish.sh.

10. Устройство читалки

Ключевые файлы: reader/index.html, reader/js/reader.js, reader/js/fbreader-ui.js, books/shared/drive-sync.js.

Поток запуска:

  • Берется book_id из ?id или hash.
  • Открывается путь /books/content/<id>/ как directory EPUB.
  • Запускается ePubReader(bookPath, opts).
  • Восстанавливается позиция (CFI) из localStorage.
  • Инициализируется UI (панели, overlays, TOC/bookmarks, mobile/desktop).

Слои:

  • epub.js — рендер-движок EPUB.
  • reader.js — оркестрация рендеров, page map, счетчик страниц.
  • fbreader-ui.js — UI-обвязка, жесты/кнопки, TTS.
  • drive-sync.js — синхронизация пользовательского состояния в Google Drive.

Функциональные блоки:

  • TOC/навигация по CFI/секциям.
  • Global page map + fallback guards (стабильность page counter).
  • TTS (speech synthesis, resume/stop, подсветка слова).
  • Shared notes через worker API: /books/api/notes-share* и алиасы /api/ns*.
  • Перевод через worker API: /books/api/translate.

Критичные зависимости для reader:

  • Наличие валидной структуры книги в content/<id>.
  • Доступность worker endpoint'ов translate/notes.
  • Синхронность версий reader.js, fbreader-ui.js, CSS.

Onboarding hints в reader:

  • Hint-логика зависит от entry query param.
  • Показывать hint только для entry=catalog; не показывать для entry=hero и entry=mybooks.
  • Текущий hint появляется через 1500ms после открытия книги и расположен в нижней центральной части viewport.
  • Текст: touch — Swipe or tap edges to turn the page →, desktop — Click on edges to turn the page →.
  • Hint автоматически скрывается через 4000ms после появления или раньше, если пользователь сделал первый реальный page turn.
  • Для определения первого пользовательского перелистывания используется существующее reader navigation событие rendition.on("relocated", ...); initial relocation на загрузке книги не считается turn.

12. Статистика / PostHog

Для reader.pub/books/ и reader.pub/books/reader/ подключен PostHog Cloud.

Важно:

  • боевой route reader.pub/books* обслуживается отдельным worker reader-books-router;
  • одного деплоя Pages-проекта reader-books недостаточно, если изменения должны сразу появиться на production URL reader.pub/books/;
  • при изменениях, влияющих на боевой reader.pub/books*, нужно учитывать production worker/router слой.

Что считается:

  • pageview каталога /books/;
  • pageview reader /books/reader/;
  • autocapture кликов в каталоге и в reader;
  • событие book_open при открытии книги.

Где смотреть:

  • Web analytics -> путь /books/ для роста каталога;
  • Web analytics -> путь /books/reader/ для роста просмотров страниц книг;
  • Activity -> autocaptured click events для кликов;
  • Activity или Insights -> book_open для факта открытия книги.

Переменные окружения: READERPUB_POSTHOG_ENABLED, READERPUB_POSTHOG_KEY, READERPUB_POSTHOG_HOST.

Operational project instructions now live in the root context files and the kept tooling READMEs.
